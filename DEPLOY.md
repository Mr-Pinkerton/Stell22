# Чек-лист деплоя Stell22

Стек: **Next.js 16 (webpack) + PostgreSQL (Prisma) + Node-рантайм**.
Важно: `proxy.ts` и серверные экшены с Prisma работают **только на Node.js**
(не Edge). Пересчёт себестоимости использует очередь в памяти
(`src/server/cost-queue.ts`) — корректно **только при одном инстансе**.

Легенда: `[ ]` — сделать перед первым продом · `[~]` — желательно ·
`[!]` — критично (данные/безопасность).

---

## 0. Модель запуска (решить заранее)
- [ ] Один инстанс приложения (из-за очереди в памяти). Горизонтальное
      масштабирование — только после переноса очереди в Redis/воркер.
- [ ] Домен + HTTPS обязателен (cookie сессии в проде помечается `secure`,
      по HTTP вход работать не будет).
- [ ] Определить, нужны ли на старте внешние интеграции (Ozon/WB, банк,
      термопринтер). Сейчас это заглушки — если нужны, это отдельная работа.

## 1. Секреты и переменные окружения `[!]`
Задать в окружении прод-сервера (НЕ коммитить, не класть в образ):
- [!] `SESSION_SECRET` — случайный, ≥32 байт. Сгенерировать:
      `openssl rand -base64 32`. Смена секрета разлогинит всех.
- [!] `DATABASE_URL` — строка подключения к боевому PostgreSQL.
- [ ] `NODE_ENV=production` (включает `secure`-cookie; выставляется автоматически
      при `next start`, но проверить).
- [ ] `SEED_ADMIN_PASSWORD` — НЕ нужен в проде (сид не запускаем). Пароль
      админа задаётся отдельно (см. п.5).
- [ ] Проверить, что `.env` в `.gitignore` (коммитится только `.env.example`).

## 2. База данных `[!]`
- [!] Поднять управляемый PostgreSQL (или свой инстанс) с автоматическими
      **бэкапами** и хранением истории. Данные себестоимости — суть системы.
- [ ] Применить миграции боевой командой (создаёт схему, НЕ трогает данные):
      ```bash
      npx prisma migrate deploy
      ```
- [!] **НЕ запускать** `npm run db:seed` / `prisma migrate reset` в проде —
      они стирают данные.
- [ ] Проверить восстановление из бэкапа (тестовый restore) до запуска.

## 3. Сборка и запуск
- [ ] Установка зависимостей: `npm ci`.
- [ ] Сгенерировать клиент Prisma: `npx prisma generate`.
- [ ] Прод-сборка: `npm run build` (должна пройти без ошибок).
- [ ] Запуск: `npm run start` (Next в Node-режиме).
- [ ] Менеджер процессов/супервизор (systemd / PM2 / контейнер с restart policy),
      автозапуск при перезагрузке сервера.
- [ ] Порядок деплоя: `migrate deploy` → `build` → рестарт приложения.

## 4. Сеть и HTTPS `[!]`
- [!] Обратный прокси (Nginx/Caddy/облачный LB) с TLS-сертификатом (Let's Encrypt).
- [ ] Проксировать на порт Next (`3000` по умолчанию), передавать заголовки
      `X-Forwarded-Proto`/`Host`.
- [ ] Закрыть прямой доступ к порту приложения и к порту PostgreSQL извне.

## 5. Учётные записи и доступ `[!]`
- [!] Сменить пароль администратора на стойкий:
      ```bash
      npx tsx scripts/set-admin-password.ts "НОВЫЙ_СТОЙКИЙ_ПАРОЛЬ"
      ```
      (скрипт делает upsert `admin@stell22.local`, данные не стирает).
- [ ] Проверить вход `/login` и что `/dashboard` без сессии редиректит на `/login`.
- [ ] Решить доступ к `/terminal` (сейчас публичный, вход по PIN сотрудника) —
      при необходимости ограничить по сети/VPN.

## 6. Безопасность (доработать) `[~]`
- [~] Rate-limit / блокировка после N неудачных попыток на `/login`
      (защита от перебора). Сейчас отсутствует.
- [~] Мониторинг ошибок (Sentry или аналог) и сбор логов приложения.
- [~] Health-check эндпоинт для супервизора/LB.
- [~] Восстановление пароля по почте — нужен SMTP (в роадмапе отложено).

## 7. Проверка после деплоя (smoke)
- [ ] `/login` открывается по HTTPS, вход проходит, кнопка «Выйти» работает.
- [ ] Ключевые разделы грузятся: дашборд, закупки, номенклатура, производство,
      отчёты, финансы.
- [ ] Создать тестовую партию → произвести → отчёт по себестоимости считается.
- [ ] Колокольчик уведомлений отображает события.
- [ ] Часовой пояс дат корректен (UTC+3, Europe/Moscow).

## 8. Эксплуатация `[~]`
- [~] Регламент бэкапов БД + проверка восстановления.
- [~] План обновлений: `git pull` → `npm ci` → `prisma generate` →
      `migrate deploy` → `build` → рестарт.
- [~] При переходе на несколько инстансов — вынести очередь пересчёта
      себестоимости в Redis/воркер (иначе двойные/потерянные пересчёты).
