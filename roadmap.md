# Roadmap — система учёта производства и себестоимости

Пошаговый план от нуля до полностью работающего продукта.
**Главная цель:** точный и непротиворечивый расчёт себестоимости изделия.

> Подход: **UI-first**. Сначала полный кликабельный прототип всех экранов на
> моках, затем оживление бэкендом по модулям. Движок расчёта себестоимости
> валидируем параллельно (чистые функции + тесты).
>
> На этом этапе НЕ рассматриваем деплой.

**Стек:** Next.js + TypeScript + PostgreSQL (Prisma). Терминал — онлайн-only.
Внешние интеграции (Ozon/WB, банк, принтер) на старте — заглушки.

**Источники истины:**
- Логика и интерфейс — `Описание проекта v2.txt`
- Модель данных — `prisma/schema.prisma`

## Отложенные задачи / техдолг

- [ ] **Склад → вкладка «Детали»: завышение «ожидает присадки».** В модели
      «поток через заготовки» идентичность детали назначается на присадке, а
      заготовки — общий пул по длине/типу/сорту. Из-за этого в складской вкладке
      «Детали» показатель «ожидает присадки» задваивает заготовки:
      - одни и те же заготовки попадают в pending каждой детали своей длины;
      - у детали с двумя присадками считаются дважды (`torcev + plosk`).
      Источник — `buildStockSnapshot` (`src/lib/detail-stock.ts`) + сборка
      `detailRows` в `getWarehouseStock` (`src/server/warehouse.ts`). На расчёт
      себестоимости и контроль остатков НЕ влияет (списания атомарны по
      `BlankStock`/`DetailStock`) — это только отображение «потенциала».
      Вариант правки: в «Деталях» показывать лишь готовые (`detailsReady`), а
      заготовки — отдельной секцией (уже есть `blanks`). Обсудить UX перед правкой.
