## Приоритет 1 — Себестоимость (ядро)

- [x] **A2** — валидация/фолбэк при `P1=P2=0`, чтобы материал не обнулялся (`src/lib/cost.ts:88-104`) + unit-тест
  - _Если у партии не заполнили цены сортов (нули), система решает «древесина бесплатная» и пишет 0₽ в себестоимость. Изделие выглядит дешевле, чем есть._
- [x] ⚠ **A1** — РЕШЕНО (Вариант 3): узаконить WAC-по-породе для изделия в v2 + правиле cost-integrity (цены партий стабильны; FIFO-слои не нужны). Уровень детали остаётся по партии. FIFO — backlog, если цены станут волатильны (`src/lib/cost-report.ts:271-294`, `Описание проекта v2.txt` §1)
  - _Материал изделия считается по средней цене породы, а не по конкретной партии, из которой деталь сделали. Две партии дуба (дорогая и дешёвая) — себестоимость выйдет неверной. Заготовка «забывает» свою партию._
- [x] **A3** — `getCostReport` читает `BatchCost` (FINAL) при `frozenAt`, live только для открытых (`src/server/cost.ts:166-204`)
  - _При закрытии партии итог замораживается (снимок), но отчёт этот снимок игнорирует и каждый раз пересчитывает заново. Два разных ответа на один вопрос._
- [x] **A4** — статус FINAL из `frozenAt`/`BatchCost.status`, а не из `ARCHIVED` (`src/lib/cost-report.ts:336`)
  - _Табличка «Итог/Окончательно» показывается по факту «партия в архиве», хотя цифры ещё предварительные и пересчитываются. Пользователь видит ложное «окончательно»._
- [x] ⚠ **A5** — согласовать заморозку: РЕШЕНО оставить код (`paid+closed`), текст v2 приведён к коду (`src/server/cost.ts:330-350`, `Описание проекта v2.txt`)
  - _Спека: замораживать при закрытии партии. Код: ещё и когда вся торцовка оплачена. Рассогласование — надо выбрать правильный вариант и свести документ с кодом._
- [x] **A6** — «работа» в отчёте из факта операций (сделка исполнителя + доля почасовой смены), а не среднее по справочнику (`src/lib/payroll.ts actualProductionRates`, `src/server/cost.ts`)
  - _Оплата труда в себестоимости берётся как средняя ставка из справочника, а не сколько реально начислили за операции. При разных расценках цифра «работа» неверная._

## Приоритет 2 — Производство / склад

- [x] **A7** — серверная валидация длины заготовок ≤ взятые рейки в `submitTorcovka` (`src/server/terminal.ts:209-286`)
  - _Интерфейс не даёт нарезать заготовок больше, чем взято реек, но вызов сервера напрямую этой проверки не делает. Заготовки из воздуха → ломается отход и цена за м³._
- [x] **A8** — та же проверка бюджета реек при правке TORCOVKA вверх (`src/server/production.ts:316-334`)
  - _Если админ правит количество торцовки вручную вверх — та же дыра: можно превысить реально взятые рейки._
- [x] **A9** — инвентаризация DETAIL: все ready-корзины детали сводятся к факту (канон=actual, лишние ready→0), НЗП/заготовки не трогаем (`src/lib/detail-stock.ts normalizeReadyBuckets`, `src/server/warehouse.ts`)
  - _Частично готовые детали (часть отверстий просверлена) лежат в промежуточных корзинах. При инвентаризации они не обнуляются → на складе числится больше деталей, чем есть._
- [x] **A10** — оценка недостачи по себестоимости («Потеря ГП»): `conductInventory` пишет `InventoryLine.deviationSum` = откл × unitCost (изделие=полная, деталь=материал+работа WAC, крепёж=цена); недостача<0=убыток, излишек>0 справочно; в отход НЕ идёт. Сырьё в этой инвентаризации не участвует. Единый `getUnitCostSnapshot()` через `loadCostContext` (переиспользование движка, шаг к A15) (`src/server/cost.ts`, `src/server/warehouse.ts`)
  - _При нехватке надо различать: не хватило сырья = отход, не хватило готовой продукции = убыток «Потеря ГП». Теперь недостача ГП/деталей/крепежа оценивается по себестоимости и не попадает в процент отхода партий._

## Приоритет 3 — Накладные / ЗП / финансы

- [x] **A11** — периодный отчёт себестоимости: `getPeriodOverhead(period)` + `getCostReport(period)`, охват по периоду, оценка ₽/м³ по полному производству (`src/server/cost.ts`, `src/lib/report-period.ts`, `src/lib/dates.ts`)
  - _Накладные (аренда, свет) раскидываются на продукцию за всё время, без фильтра дат. На единицу продукции падает то слишком много, то слишком мало._
- [x] **A12** — дата-часть: период прокинут во ВСЕ вкладки (закупки по `purchaseDate`, отход/ЗП по `workDate`/`paidAt`, себестоимость); неделя пт–чт в URL для ЗП; кнопка «Выплатить» (выплата = весь неоплаченный период, не трогается); границы периода в UTC+3 (`lib/dates.ts startOfBusinessDay/endOfDay`)
  - _Зарплатная неделя (пт–чт) и фильтры по датам нарисованы, но не подключены. «Выплатить за неделю» гасит вообще все невыплаченные начисления, фильтр дат — картинка._
- [x] **A13** — единое правило `confirmed` (карантин) для cost/deal/накладных: `sumConfirmedExpense` (расход сделки только по подтверждённым счетам) в `syncDeal`/`syncBatchTotalCost`/`loadDeal`; `account.confirmed` в `getPeriodOverhead`. Операция с неподтверждённого счёта импорта больше не меняет `totalCost` партии / `deal.total` / накладные до подтверждения счёта (`src/lib/deal-cost.ts`, `src/server/finance.ts`, `src/server/cost.ts`)
  - _Импорт выписки на авто-созданный (неподтверждённый) счёт сразу двигал себестоимость партии и накладные, хотя в ДДС/KPI такие операции скрыты до подтверждения. Теперь правило единое._
  - _Суммы с неподтверждённого счёта (карантин) не должны влиять на расчёты, но стоимость партии может измениться от денег, которых ещё нет в движении._
